What you can verify in 3 minutes

  1. Open the live app. Try Switchback → The app runs in demo mode if no ANTHROPIC_API_KEY is configured; in production it generates a real plan from a plain-language goal in 30–60s.
  2. Generate or load a plan. Type a goal (e.g. "UTMB CCC in August 2026, currently running ~45 km/week") and submit, or tap Load sample for an instant 4-week excerpt. The plan is structured: phases (base / build / peak / taper), weekly mileage caps, typed workouts (easy, long, intervals, hills, recovery, rest), and per-workout coaching notes.
  3. Download a .FIT workout. Every non-rest workout has a .FIT button. The encoded file is a Garmin structured workout (file type 5 = WORKOUT, sport = RUNNING, with warmup / main / cooldown steps) — not an activity file.
  4. Verify in Garmin Connect. Open Garmin Connect → Workouts, click Create a Workout → Import, and select the downloaded .FIT. The workout should appear with the correct structure, duration, and step types. Send it to a paired watch to complete the round trip.
Why this matters for our request We can already shape valid Garmin workout files end-to-end. The Training API is what turns this from a manual import flow (the alpha workaround) into an automatic generate → push → sync experience — the same path TrainingPeaks and other approved partners use.

What we're asking for

Access to the Training API to push structured workouts into a connected user's Garmin Connect calendar. One weekly batch per user (Sunday evening) plus occasional ad-hoc adjustments when a plan is adapted. Volume forecast, data-handling posture, and safety/liability language are in the application document we've prepared — happy to send on request, or read the public draft at docs/garmin-application.md.

Read access to the Activity / Health API would also benefit our weekly adaptation loop but is not a launch blocker. We're happy to scope it in the same conversation or defer to a follow-up.

How we handle user data

Garmin OAuth tokens only — we never request or store Garmin passwords. Tokens are encrypted at rest with a per-user key in an EU-region Postgres database. Disconnect deletes tokens the same request. Data retention, processors, and DPAs are enumerated in our Privacy Policy. The medical / liability posture is in our Terms of Service §6.
